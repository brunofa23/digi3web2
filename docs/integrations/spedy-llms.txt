# Spedy API — Documentação para Integração

> Documentação otimizada para uso por assistentes de IA e ferramentas de desenvolvimento.
> Idioma: Português (pt-BR)
> Base URL: https://api.spedy.com.br/v1

---

## O que é a Spedy API

A Spedy é uma plataforma SaaS de automação fiscal brasileira. A API REST permite emitir e gerenciar notas fiscais de produto (NF-e), serviço (NFS-e) e consumidor (NFC-e) em todo o Brasil, integrando diretamente com a SEFAZ estadual e prefeituras municipais.

Casos de uso principais:
- SaaS e infoprodutores que precisam emitir NFS-e automaticamente após cada venda
- E-commerces e varejistas que emitem NF-e por pedido
- ERPs que precisam de emissão programática com controle tributário completo
- Contabilidades gerenciando múltiplos CNPJs

## Ambientes

| Ambiente | API Base URL | Backoffice |
|---|---|---|
| Produção | https://api.spedy.com.br/v1 | https://app.spedy.com.br |
| Sandbox | https://sandbox-api.spedy.com.br/v1 | https://sandbox-app.spedy.com.br |

Sandbox é completamente isolado: banco de dados independente, exige novo cadastro com Plano Desenvolvedor (gratuito). Uma chave de produção **não** funciona no sandbox e vice-versa.

## Autenticação

Toda requisição deve incluir o header:

```
X-Api-Key: SUA_CHAVE_AQUI
```

A autenticação é por empresa (CNPJ). Cada empresa cadastrada possui sua própria chave de API.

### Hierarquia de Permissões

- **Empresa Owner** (empresa principal da conta): pode criar, editar e excluir outras empresas, além de emitir notas.
- **Empresas secundárias**: podem apenas emitir notas. Não gerenciam outras empresas.

A chave da nova empresa é retornada no campo `apiCredentials.apiKey` ao criar via `POST /v1/companies`.

---

## Rate Limit

- **60 requisições por minuto**
- **5 requisições por segundo**
- Aplicado por chave de API (por empresa)
- Contas com múltiplos CNPJs têm limite independente por chave

Se ultrapassado: `429 Too Many Requests`

Headers retornados em cada requisição:

| Header | Descrição |
|---|---|
| `x-rate-limit-limit` | Janela de tempo usada (ex: `1m`) |
| `x-rate-limit-remaining` | Requisições restantes |
| `x-rate-limit-reset` | Data/hora UTC em que o limite reseta |

## Formato de Resposta

A API retorna os objetos **diretamente**, sem envelope de wrapper.

Para objetos simples:
```json
{
  "id": "uuid",
  "status": "authorized",
  ...
}
```

Para listas paginadas:
```json
{
  "items": [...],
  "totalCount": 150,
  "pageCount": 8,
  "pageSize": 20,
  "hasNext": true
}
```

Para operações que retornam booleano (cancelar, excluir):
```json
{ "success": true }
```

Parâmetros de paginação: `?page=1&pageSize=20`

## Status e processingDetail

### status

O campo `status` representa o estado atual da nota fiscal e é controlado exclusivamente pela Spedy. **Não deve ser enviado nas requisições.** Use-o para acompanhar o resultado das operações.

Valores: `enqueued`, `authorized`, `rejected`, `canceled`, `denied`, etc. (ver tabela InvoiceStatus na seção de referência).

### processingDetail

O objeto `processingDetail` contém o resultado do **último processamento** executado na nota. Toda operação (emissão, cancelamento, carta de correção) atualiza esse objeto. Os campos são:

| Campo | Descrição |
|---|---|
| `status` | `success`, `processing` ou `failed` |
| `message` | Mensagem descritiva do resultado |
| `code` | Código retornado pela autoridade fiscal |

Quando uma nota é rejeitada (`status: "rejected"`), os detalhes da rejeição ficam em `processingDetail.message` e `processingDetail.code`. Rejeições podem vir diretamente da SEFAZ/Prefeitura ou da validação prévia da Spedy — que utiliza o schema fornecido pela autoridade fiscal para validar os dados antes do envio. Quando a rejeição é da Spedy, o código em `processingDetail.code` inicia com o prefixo `SPD`. Em ambos os casos, o formato é idêntico: nota com `status: "rejected"` e detalhes em `processingDetail`.

**Exemplo de nota rejeitada:**
```json
{
  "status": "rejected",
  "processingDetail": {
    "status": "success",
    "message": "Código do município do prestador não encontrado",
    "code": "E125"
  }
}
```

**Nota sobre `processingDetail.status`:** O valor `success` indica que o processamento assíncrono foi executado com sucesso — **não** que a nota foi autorizada. Mesmo uma nota rejeitada terá `processingDetail.status: "success"` porque o evento na fila foi processado corretamente. O valor `failed` ocorre apenas quando há falha interna no processamento do evento na fila (erro de infraestrutura, timeout interno, etc.), independentemente do resultado fiscal.

## Fluxo de Emissão (Assíncrono)

A emissão é **assíncrona**. O cliente envia a requisição, a nota entra em fila e é processada em background.

```
Você → POST /v1/orders (ou /product-invoices ou /service-invoices)
         ↓
     status: enqueued
         ↓
     Spedy transmite para SEFAZ / Prefeitura
         ↓
     Autoridade fiscal processa
         ↓
     Spedy atualiza status da nota
         ↓
     Webhook notifica seu sistema
```

Tempo médio de autorização: menos de 10 segundos em condições normais.

**Emissão por venda (`/orders`):** o enfileiramento depende do `autoIssueMode` da venda. A nota só entra na fila quando o modo permite a emissão — `immediately` (ao criar a venda), `afterPayment` (quando a venda muda para `approved`) ou `afterWarrency` (no dia seguinte ao fim da garantia). Com `disabled`, a venda é registrada mas nenhuma nota é emitida automaticamente. Ver seção "Gatilhos de Emissão (autoIssueMode)".

**Recomendação:** use Webhooks para receber atualizações de status. Evite polling.

### Polling como Alternativa

Quando webhooks não forem viáveis, use polling:

1. Após emitir, consulte `GET /v1/service-invoices/{id}` ou `GET /v1/product-invoices/{id}`
2. Verifique o campo `status` — `created` significa que a nota aguarda o disparo da emissão; `enqueued` indica que está na fila de processamento. Ambos são estados intermediários antes da autorização ou rejeição
3. Intervalo recomendado: 5 a 10 segundos entre consultas
4. Tempo médio de autorização: menos de 10 segundos
5. Pare quando `status` mudar para um estado final: `authorized`, `rejected`, `canceled` ou `denied`

**Para polling, use sempre o `GET`** — ele retorna o `status` atual sem acionar a autoridade fiscal. **Não** use o `POST .../check-status` para acompanhar a emissão: ele faz uma consulta síncrona ao fisco, é reservado para reconciliação/destrave de notas já enviadas, e retorna **400** enquanto a nota está `enqueued` ou `created`.

## Contingência

Quando a **SEFAZ está temporariamente indisponível** (instabilidade, pico de demanda ou manutenção), a Spedy emite por um caminho de contingência previsto na legislação, mantendo a operação. Aplica-se a **NF-e** e **NFC-e** (SEFAZ); não se aplica a NFS-e.

**A contingência é automática.** A Spedy monitora a disponibilidade da SEFAZ e decide sozinha quando entrar e sair. Não há como forçá-la por requisição nem escolher o modo por nota — sua chamada de emissão não muda.

**NF-e (SVC):** a autorização passa a um ambiente alternativo da própria SEFAZ (SVC — Sistema Virtual de Contingência). É síncrona e transparente — o `status` segue normalmente para `authorized` e a DANFE sai identificada como emitida em contingência.

**NFC-e (offline):** a nota é **válida no momento da emissão** (`status: inContingent`), com DANFCE e XML já disponíveis. Quando a SEFAZ volta, a Spedy transmite a nota automaticamente para autorização definitiva (`authorized`) ou rejeição (`rejected`). Enquanto estiver `inContingent`, o cancelamento fica indisponível.

> **Pré-requisito da NFC-e offline:** exige `consumerInvoice.allowOfflineContingency = true` nas configurações da empresa (`PUT /v1/companies/{id}/settings`), com `tokenId`/`csc` válidos. Sem essa habilitação, a NFC-e segue o fluxo normal de retentativa. A habilitação apenas autoriza o uso da contingência offline — quando entrar em contingência continua sendo automático da Spedy.

Acompanhe pelo `status` (ver InvoiceStatus) e pelos webhooks: `invoice.contingency` na entrada em contingência, e `invoice.authorized`/`invoice.rejected` na regularização.

## Modos de Emissão

### Modo Simplificado — POST /v1/orders

Ideal para SaaS, infoprodutores, e-commerce com tributação padronizada.

Vantagens:
- Não requer CFOP, NCM nem cálculo de impostos
- Cliente e produto são criados automaticamente se não existirem
- Toda a configuração tributária fica no backoffice da Spedy

Use quando: tributação é uniforme e o cliente final não precisa controlar detalhes fiscais por produto.

### Modo Completo — POST /v1/product-invoices, /v1/service-invoices ou /v1/consumer-invoices

Indicado para:
- Tributação que varia por produto (diferentes CFOPs, NCMs, alíquotas)
- ERPs e sistemas de varejo
- Cenários em que o cliente não usa o backoffice da Spedy para configuração

Requer envio completo dos dados tributários (ICMS, PIS, COFINS para NF-e/NFC-e; ISS, retenções para NFS-e).

## integrationId — Identificador de Integração

Campo **fortemente recomendado** disponível nos endpoints `POST /v1/product-invoices`, `POST /v1/service-invoices` e `POST /v1/consumer-invoices` (máx. 36 caracteres).

| Função | Descrição |
|---|---|
| **Associação** | Vincula a nota gerada pela Spedy a um ID do sistema do cliente (ex: ID do pedido) |
| **Idempotência** | Um segundo POST com o mesmo `integrationId` atualiza a nota existente em vez de criar uma nova — protege contra duplicidade por timeout ou retry |
| **Correção de rejeitada** | Para corrigir uma nota rejeitada, reenvie o POST com os dados corrigidos e o mesmo `integrationId` — o sistema atualiza a nota, sem necessidade de deletar |

**Fortemente recomendado** em todas as integrações. Use o ID natural da transação no sistema de origem (ex: ID do pedido, ID da ordem de serviço, número da venda). Além de evitar duplicidade, facilita rastreamento e correção de notas rejeitadas.

## Gatilhos de Emissão (autoIssueMode)

Campo do endpoint `/v1/orders` que controla quando a nota é emitida.

| Valor | Comportamento |
|---|---|
| `immediately` | Emite ao criar a venda (padrão) |
| `disabled` | Não emite automaticamente |
| `afterPayment` | Emite ao status da venda mudar para `approved` |
| `afterWarrency` | Emite no dia seguinte ao fim da garantia (`warrantyDate`) |

### Exemplo prático: boleto bancário

```
1. POST /v1/orders
   → status: "awaitingPayment"
   → autoIssueMode: "afterPayment"

   A nota é criada internamente mas NÃO enfileirada.

2. Boleto compensado → PUT /v1/orders/{id}/status
   → { "status": "approved" }

   Spedy detecta o gatilho e emite automaticamente.
   Nenhuma outra chamada é necessária.
```

## Conceitos Importantes

### Venda ≠ Nota

A venda (Order) é usada como gatilho para emissão da nota fiscal. Uma venda pode gerar uma ou mais notas. O `autoIssueMode` define quando a emissão acontece.

### Split Fiscal

Uma venda pode gerar NF-e + NFS-e simultaneamente (ex: produto físico + instalação). Requer habilitação no backoffice.

### Cálculo de Valores

```
amount da venda = soma(item.amount) + freightAmount - discountAmount
item.amount     = quantity × price - item.discountAmount
```

A soma dos `item.amount` deve ser igual ao `amount` da venda.

### Cancelamento

`DELETE /v1/orders/{id}/cancel` tenta cancelar a venda e as notas associadas.

- NF-e dentro do prazo legal → cancela normalmente
- NF-e fora do prazo → pode gerar NF-e de devolução automática
- NFS-e fora do prazo → depende da prefeitura, em geral não é possível

**Importante:** cancelar a venda não garante cancelamento da nota. Sempre verifique o status da nota após o cancelamento.

Todos os objetos de cidade na API (endereço do cliente, localização do serviço, etc.) podem ser preenchidos de duas formas:

### Opção 1 — Por código IBGE (recomendado quando disponível)

```json
{
  "city": {
    "code": "3550308"
  }
}
```

Mais assertivo: não depende de matching de nome. Use quando o código IBGE estiver disponível no sistema de origem.

### Opção 2 — Por nome e estado

```json
{
  "city": {
    "name": "São Paulo",
    "state": "SP"
```

Alternativa quando o código IBGE não está disponível. A Spedy resolve automaticamente o município pelo nome e estado.

## Webhooks

A Spedy envia HTTP POST para sua URL quando eventos ocorrem nas notas fiscais.

### Eventos Disponíveis

| Evento | Descrição |
|---|---|
| `invoice.authorized` | Nota fiscal autorizada |
| `invoice.canceled` | Nota fiscal cancelada |
| `invoice.contingency` | Nota fiscal emitida em contingência, aguardando autorização definitiva |
| `invoice.rejected` | Nota fiscal rejeitada |
| `invoice.status_changed` | Status alterado (qualquer mudança) |

### Schema do Payload

```json
{
  "id": "uuid-do-evento",
  "event": "invoice.status_changed",
  "data": { ... }
}
```

### Política de Retry

Quando a entrega de um webhook falha (endpoint indisponível, timeout, HTTP 5xx), a Spedy reenvia automaticamente com intervalos crescentes:

| Tentativa | Intervalo |
|---|---|
| 1ª | 5 minutos |
| 2ª | 30 minutos |
| 3ª | 1 hora |
| 4ª | 4 horas |
| 5ª | 16 horas |

Após 5 falhas consecutivas, o webhook é **desabilitado automaticamente**. Para reativá-lo, use o endpoint `PUT /v1/webhooks/{id}/enable`.

**Dica:** monitore seus webhooks e implemente logs para identificar falhas de entrega rapidamente.

- `password`: senha do certificado

#### PUT /v1/companies/{id}/settings — Configurar Empresa

Configurações de emissão: série padrão, número inicial de NF-e/NFC-e, sequência de RPS para NFS-e, ambiente (produção/homologação), configurações tributárias padrão.

O corpo inclui os objetos `productInvoice` (NF-e), `consumerInvoice` (NFC-e) e `serviceInvoice` (NFS-e). Para habilitar NFC-e, configure `consumerInvoice.enabled = true` e forneça `consumerInvoice.tokenId` e `consumerInvoice.csc` (obtidos no portal da SEFAZ do estado). Para permitir a emissão em contingência offline da NFC-e quando a SEFAZ estiver indisponível, defina `consumerInvoice.allowOfflineContingency = true` (exige `tokenId`/`csc` válidos) — ver seção "Contingência".

O objeto `general` inclui o **responsável técnico** (`technicalResponsible`) da NF-e/NFC-e. Quem não configurar continua com a Spedy como responsável técnico (comportamento padrão). Configurado na empresa principal (owner), vale para todas as empresas da conta; pode ser sobrescrito em uma empresa específica.

```json
{
  "general": {
    "technicalResponsible": {
      "federalTaxNumber": "12ABC34501DE35",
      "contactName": "ACME Software",
      "email": "contato@acme.com.br",
      "phone": "11999998888",
      "csrts": [
        { "state": "PR", "environmentType": "production", "idCsrt": 1, "csrt": "SEU-CODIGO-CSRT" }
      ]
    }
  }
}
```

`federalTaxNumber` (CNPJ, sem máscara), `contactName`, `email` e `phone` são obrigatórios em conjunto. O `csrts` (Código de Segurança do Responsável Técnico) só é aceito para estados que o exigem — hoje apenas o Paraná (`PR`); em PR com responsável próprio mas sem CSRT cadastrado, a nota sai com a Spedy como responsável (a emissão nunca trava). O `csrt` é **write-only**: aceito na configuração, nunca retornado em consultas.

#### GET /v1/companies/{id}/settings — Obter Configurações da Empresa

Retorna as configurações da empresa na mesma estrutura aceita pelo PUT. Segredos são **write-only** e nunca retornam: `serviceInvoice.password`, `consumerInvoice.csc` e `technicalResponsible.csrts[].csrt`.

### Vendas — Fluxo Simplificado

Use `/v1/orders` quando a tributação é padronizada e configurada no backoffice da Spedy. A nota é emitida automaticamente com os dados cadastrais da empresa.

> **Origem dos dados tributários (Orders):** Neste modo, toda a configuração tributária (códigos de serviço, alíquotas, CFOP, NCM) é gerenciada pelo backoffice da Spedy. O sistema do cliente não precisa enviar dados tributários na API — apenas os dados comerciais da venda (cliente, itens, valores).

#### POST /v1/orders — Criar Venda

**Exemplo 1: NFS-e, emissão imediata, pagamento por PIX**

```json
{
  "transactionId": "TXN-20240115-001",
  "date": "2024-01-15T10:00:00",
  "amount": 297.00,
  "autoIssueMode": "immediately",
  "status": "approved",
  "paymentMethod": "pix",
  "sendEmailToCustomer": true,
  "customer": {
    "name": "Maria Souza",
    "federalTaxNumber": "12345678901",
    "email": "maria@email.com.br",
    "address": {
```

### NF-e — Nota Fiscal de Produto

Use `/v1/product-invoices` quando precisar controlar explicitamente a tributação (CFOP, NCM, ICMS, PIS, COFINS).

Consulte os campos do item com `[tag_xml]` para referência cruzada com o leiaute SEFAZ.

> **Origem dos dados tributários (NF-e):** Os campos tributários (`cfop`, `ncm`, `cst`, alíquotas de ICMS/PIS/COFINS) devem ser enviados pelo sistema do cliente. Em integrações reais, esses dados tipicamente vêm de:
> - **Cadastro de produtos**: cada produto tem NCM, CFOP e CST próprios
> - **Regras específicas**: alíquotas e outros campos variando por natureza de operação, tipo do cliente (física/jurídica), etc. — nesse caso, implemente uma tabela de regras tributárias
> - **Configuração fixa da empresa**: quando toda a operação usa a mesma tributação
>
> Defina a abordagem antes de implementar. Para impostos calculados (ICMS, PIS, COFINS), é possível informar apenas a alíquota (`rate`) — o sistema calcula o valor automaticamente. Se preferir, o valor calculado também pode ser enviado.

O campo `integrationId` (máx. 36 chars) é recomendado em todas as integrações: garante idempotência (evita duplicatas em retries) e permite corrigir nota rejeitada reenviando o POST com o mesmo `integrationId` e os dados corrigidos — sem necessidade de deletar a nota anterior.

### NFS-e — Nota Fiscal de Serviço

Use `/v1/service-invoices` para emissão de notas de serviço (prefeitura). Antes de emitir, verifique se o município está integrado via `GET /v1/service-invoices/cities`.

> **Origem dos dados tributários (NFS-e):** Os campos `federalServiceCode`, `cityServiceCode`, `nbsCode` e `taxationType` devem ser enviados pelo sistema do cliente. Em integrações reais, esses dados tipicamente vêm de:
> - **Cadastro de serviços**: cada serviço tem código LC 116/03, código NBS e alíquota ISS próprios
> - **Regras específicas**: alíquotas e outros campos variando por localidade do cliente, tipo de pessoa (física/jurídica), valor da NF, etc. — nesse caso, implemente uma tabela de regras tributárias
> - **Configuração fixa da empresa**: quando toda a operação usa a mesma tributação (ex: empresa de TI que emite apenas NFS-e com código 1.07 e ISS 5%)
>
> **Importante:** Antes de integrar, consulte `GET /v1/service-invoices/cities` para verificar quais campos são obrigatórios no município da empresa emissora. Cada cidade retorna um objeto `provider.options` que indica se o município exige `federalServiceCode`, `nbsCode`, numeração de RPS, série, lote, entre outros. Esse endpoint é essencial para montar corretamente o request de emissão.
>
> Defina a abordagem antes de implementar. Para o ISS e demais impostos (PIS, COFINS, IR, CSLL, INSS), é possível informar apenas a alíquota (`rate`) — o sistema calcula o valor automaticamente. Se preferir, o valor calculado também pode ser enviado.

Retorna a lista de municípios disponíveis para emissão de NFS-e com as configurações técnicas do provedor de cada cidade. **Consulte este endpoint antes de integrar um novo município** para identificar os campos obrigatórios e o tipo de autenticação exigido.

- `cityServiceCode`: código de serviço específico do município — consultar tabela da prefeitura. **Atenção:** o formato varia por município (pode ter 4, 6 ou 7 dígitos). Não assuma que é o `federalServiceCode` sem o ponto.
- `description`: discriminação dos serviços. Ser descritivo; muitos municípios exigem detalhamento
- **Dica para composição do `description`:** Componha a descrição a partir dos itens do pedido/OS. Exemplo:
  ```
  Ref. OS #0001
  - Manutenção Preventiva (Qtd: 2)
  - Consultoria Técnica (Qtd: 1)
  ```
- `issWithheld: true`: ISS retido na fonte pelo tomador. Obrigatório para alguns municípios quando o tomador é pessoa jurídica estabelecida no mesmo município
- Alíquota de ISS varia por município e tipo de serviço (geralmente entre 2% e 5%)
- Para Simples Nacional: campos de PIS, COFINS, CSLL, IR geralmente são omitidos pois já estão cobertos pelo Simples

Notas sobre `taxationType`:
- `taxationInMunicipality`: ISS tributado no município onde o serviço é prestado (prestador) — caso mais comum
- `taxationOutsideMunicipality`: ISS tributado no município do tomador (casos especiais da LC 116/03, art. 3º)
- `exemption`: isento de ISS
- `exportation`: exportação de serviços para o exterior

### NFC-e — Nota Fiscal ao Consumidor

Use `/v1/consumer-invoices` para emitir NFC-e (cupom fiscal eletrônico, modelo 65). Destinado a varejo presencial e operações com consumidor final.

Consulte os campos do item com `[tag_xml]` para referência cruzada com o leiaute SEFAZ.

**Pré-requisitos:** NFC-e deve estar com `tokenId` e `csc` configurados em (`PUT /v1/companies/{id}/settings`) (obtidos no portal da SEFAZ do estado).

> **NFC-e — todos os dados devem ser enviados via API.** Diferentemente de NF-e e NFS-e, a NFC-e não possui configuração de tributação padrão via backoffice. Todos os campos tributários (`cfop`, `ncm`, `cst`, alíquotas de ICMS/PIS/COFINS) devem ser enviados na requisição. Isso inclui:
> - `cfop`, `ncm`, `unit` e tributação completa em cada item
> - Alíquotas e bases de ICMS, PIS e COFINS
> - Forma de pagamento no campo `payments`
>
> Defina a abordagem no seu sistema antes de integrar. Para impostos calculados (ICMS, PIS, COFINS), é possível informar apenas a alíquota (`rate`) — o sistema calcula o valor automaticamente. Se preferir, o valor calculado também pode ser enviado.

O campo `integrationId` (máx. 36 chars) é recomendado em todas as integrações: garante idempotência (evita duplicatas em retries) e permite corrigir nota rejeitada reenviando o POST com o mesmo `integrationId` e os dados corrigidos — sem necessidade de deletar a nota anterior.

#### POST /v1/consumer-invoices — Criar NFC-e

Mesmos campos base da NF-e. Diferenças principais:
- `isFinalCustomer` deve ser `true` (NFC-e é sempre para consumidor final)
- `destination` é sempre `internal` (NFC-e não permite operação interestadual)
- `presenceType` deve ser `presential` (venda presencial) ou `internet` (venda online)
- `receiver.federalTaxNumber` é opcional (CPF do consumidor) — pode ser omitido para vendas anônimas
- Não há carta de correção após emissão
- QR Code é gerado automaticamente com base no `tokenId` e `csc` configurados na empresa

CFOPs para NFC-e (sempre internos):
- `5102` = venda de mercadoria adquirida de terceiros ao consumidor final
- `5405` = venda de mercadoria sujeita a ST ao consumidor final
- `5949` = outras saídas

### Webhooks

**Escopo: conta, não empresa.** Um webhook recebe eventos de todas as empresas da conta. Use o objeto `company` no payload para identificar qual empresa gerou o evento.

#### Eventos disponíveis

| Evento | Descrição |
|---|---|
| `invoice.status_changed` | Qualquer alteração de status da nota — cobre criação, autorização, rejeição, cancelamento etc. |
| `invoice.authorized` | Nota fiscal autorizada |
| `invoice.rejected` | Nota fiscal rejeitada |
| `invoice.canceled` | Nota fiscal cancelada |

Na maioria dos casos, assinar apenas `invoice.status_changed` é suficiente para cobrir todo o ciclo de vida da nota.

#### POST /v1/webhooks — Criar Webhook

```json
{
  "event": "invoice.status_changed",
  "url": "https://meusite.com.br/webhooks/spedy"
}
```

Crie um registro por evento. Para receber múltiplos eventos, crie um webhook para cada um.

## Guia de Integração Rápida

### Checklist de Setup

1. Criar conta no sandbox: https://sandbox-app.spedy.com.br (Plano Desenvolvedor, gratuito)
2. Configurar empresa no backoffice (regime tributário, certificado digital, série NF)
3. Obter chave de API: Perfil → Minha Empresa → Credenciais da API
4. Criar webhook: `POST /v1/webhooks` com `event: "invoice.status_changed"`
5. Testar emissão com `POST /v1/orders` (fluxo simplificado)
6. Ao validar, migrar para produção com nova conta e chave

### Qual Endpoint Usar?

```
Preciso emitir nota fiscal automaticamente
│
├─ Tributação uniforme (mesmo ISS/ICMS para todos os produtos)?
│   └─ SIM → POST /v1/orders
│           (configurar tributação no backoffice, sem enviar impostos na API)
│
└─ Tributação varia por produto/operação, ou cliente não usa backoffice?
    ├─ Produto físico (NF-e)           → POST /v1/product-invoices
    ├─ Serviço (NFS-e)                 → POST /v1/service-invoices
    └─ Varejo / consumidor final (NFC-e) → POST /v1/consumer-invoices
```

### Diagnóstico de Erros Comuns

| Situação | O que verificar |
|---|---|
| Nota rejeitada (status: rejected) | Verificar `processingDetail.message` e `.code` — rejeição vem da SEFAZ ou Prefeitura |
| Nota rejeitada com código SPD | Validação da Spedy usando schema da autoridade fiscal, antes do envio — verificar `processingDetail.message` |
| Nota não emitida após criar venda | Verificar `autoIssueMode` e `status` da venda |
| NFS-e não aceita pelo município | Confirmar que o município está na lista via `GET /v1/service-invoices/cities` |
| ICMS calculado errado | Conferir `baseTaxModality` (use `3` para valor da operação) e `destination` (internal/interstate) |
| ISS não calculado corretamente | Confirmar `taxationType` e se `issWithheld` corresponde ao que o município exige |
